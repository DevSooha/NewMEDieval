# 포트폴리오 기술 노트 — 안정화 라운드 사례 (2026-07-06)

버그 안정화 라운드에서 나온, 기술 면접/포트폴리오에서 설명 가치가 큰 설계 결정 4건을
"문제 → 근본 원인 → 설계 결정 → 검증" 구조로 정리한다.
상세 배경은 `BUG_STABILIZATION_SUMMARY_20260706.md` 참조.

## 사례 1. 방 데이터 이중 소스와 BFS 도달 그래프 안전망 (커밋 `7322560`)

**문제** — 특정 방에서 사망 후 재시작하면 맵이 통째로 로드되지 않는다. 평상시 플레이에서는
전혀 재현되지 않아 원인 추적이 어려웠다.

**근본 원인** — 방 시스템의 진실원(source of truth)이 둘로 갈라져 있었다.
평상시 방 이동은 `RoomData` 이웃 참조(그래프 순회)로 동작하지만, 재시작 복원은
`allMapRooms` 목록의 **좌표 선형 검색**으로 동작한다. 목록에 누락된 방(aut_3 등)은
플레이 중 정상 진입이 가능하지만, 그 안에서 재시작하는 순간 좌표 매칭이 실패해
방을 0개 스폰했다. "데이터 이중화는 반드시 한쪽이 어긋난 순간이 아니라,
어긋난 쪽을 읽는 코드 경로가 실행되는 순간 터진다"는 전형적 사례.

**설계 결정** — 데이터 보수(에디터)와 코드 안전망을 병행했다. `RoomManager.Start()`에서
RoomData 이웃 + 프리팹 `MapNode.nextRoom` 두 채널을 BFS로 순회해 도달 가능한 방을
allMapRooms에 자동 보강하고, 보강이 발생하면 "데이터가 빠졌다"는 경고를 남긴다
(경고는 세션당 1회 요약 — 로그 소음 방지, `f1f1aaa`). 매칭 실패는 디버그 플래그와
무관한 LogError로 승격해, 다음에 같은 계열 결함이 생기면 콘솔에서 즉시 드러나게 했다.

**검증** — 여러 방에서 사망 → 재시작 → 맵 정상 로드 플레이테스트 통과.

## 사례 2. 방 전환의 트랜잭션화 — 실패 시 전체 롤백 (커밋 `4a814d0`, `c8c1e6f`)

**문제** — 막힌 통로나 지면 타일이 없는 지점으로 방 전환이 시도되면, 카메라와 방 상태는
넘어갔는데 플레이어는 못 넘어가는 "갈라짐"이나 벽 바깥에 갇히는 상태가 발생.

**근본 원인** — 방 전환이 여러 단계(플레이어 이동, 카메라 스냅, 방 활성/비활성)로
이루어지는데 중간 단계 실패를 처리하지 않았다. 부분 성공 상태가 그대로 노출됐다.

**설계 결정** — 전환을 트랜잭션처럼 다뤘다. 전환 시작 시점의 상태를 보관하고,
목적지 검증(셀 일치 + `Player.IsOnWalkableGround()` 지면 판정)에 실패하면 플레이어 위치,
카메라, 방 상태를 시작 상태로 전부 되돌린다. 지면 판정은 기존
`ConstrainToWalkablePosition`의 "조용한 실패" 2경로(타일맵 미발견 시 통과,
fallback 실패 시 현위치 반환)를 명시적 판정 함수로 분리해 해결했다.

**검증** — 막힌 방향으로 전환 시도 → "전환 롤백" 경고와 함께 제자리 유지. QA 콘솔의
"플레이어 비정상 위치" 자동 감시(1초 주기)로 회귀 감지 상시화.

## 사례 3. 정밀 히트박스 설계 복구 — 태그 게이트에 막힌 자식 콜라이더 (커밋 `a78d2e9`)

**문제** — 보스 투사체 판정이 체감상 지나치게 후하다. 스프라이트를 스쳐 지나가는
투사체에 맞는다.

**근본 원인** — 플레이어에는 본체 0.6배 크기의 정밀 판정용 자식 히트박스
(`CombatTargetHitbox`)가 설계되어 있었지만, 투사체의 `OnTriggerEnter2D`가
`CompareTag("Player")` 게이트를 먼저 통과시키는데 자식 히트박스는 Untagged라
게이트를 통과하지 못했다. 결과적으로 모든 투사체가 큰 본체 캡슐에 명중 —
**정밀 히트박스가 존재하지만 단 한 번도 사용된 적 없는** 상태였다.

**설계 결정** — 판정을 임의로 좁히는 너프가 아니라 "존재하던 설계의 복구"로 접근했다.
정밀 히트박스가 있으면 본체 명중을 양보(defer)하고 히트박스 명중만 유효 처리하는
공통 판정(`IsPlayerHitCandidate`/`ShouldDeferToPreciseHitbox`)을 만들어 투사체 베이스에
적용했다. 레이/지속형 5종과 포획 기믹(Masque)은 난이도 체감이 크게 변할 수 있어
의도적으로 제외 — 밸런스 결정과 버그 수정을 분리했다.

**검증** — QA 콘솔 F4 콜라이더 시각화(빨강=정밀 히트박스, 노랑=투사체)로 명중 조건을
눈으로 확인. 플레이테스트에서 체감 판정 개선 확인.

## 사례 4. 순간이동 넉백 → 충돌 검사 기반 분할 이동 (커밋 `8cbf85a`, `1880a5f`, `2fdaaa9`)

**문제** — 문 근처에서 몬스터를 공격하면 몬스터가 문 너머나 벽 위로 순간이동하듯 튕긴다.

**근본 원인** — 몬스터 넉백 2경로(접촉 셀프 넉백, 포션 넉백)가 `rb.MovePosition` 단발
순간이동으로 구현되어 벽/문 충돌 검사가 전무했다. 흥미로운 점은 "찌르면 튕긴다"는
관측과 달리 근접 공격은 넉백을 주지 않는다는 것 — 실체는 몬스터가 플레이어와
접촉하는 순간 발동하는 셀프 넉백이었다(공격하려면 접근해야 하므로 상관관계가
인과관계처럼 보인 사례).

**설계 결정** — 프로젝트에 이미 존재하던 안전 패턴(Player 넉백의 `rb.Cast` 벽 검사)을
준용했다. 넉백을 duration 동안의 분할 이동으로 바꾸고, 각 스텝을 충돌 매트릭스 기반
`rb.Cast`로 클램프 + 방 셀 경계로 2차 클램프. 이 과정에서 시그니처에 있었지만
무시되던 `duration` 파라미터의 의도를 복구했다(받은 시간대로 분할 이동).

**검증** — 문/벽 근처 접촉·포션 넉백 플레이테스트에서 방 이탈 0건. QA 콘솔
"몬스터 방 셀 이탈" 자동 감시로 회귀 감지 상시화.

## 공통 방법론

1. **관측된 증상과 실제 원인을 분리** — 코드 추적으로 인과를 확정한 뒤에만 수정
   (사례 4의 "찌르면 튕김" 오인, 사례 1의 "평상시엔 멀쩡" 함정).
2. **수정과 정책 결정을 분리** — 난이도/조작감이 바뀌는 선택지는 밸런스 판단으로
   넘기고, 버그 수정은 "원래 설계의 복구" 범위로 한정 (사례 3).
3. **회귀를 사람 눈에 맡기지 않음** — 수정마다 QA 콘솔 자동 감시(셀 이탈, 열린 문,
   비정상 위치)를 함께 심어 다음 회귀가 스스로 드러나게 함.
4. **씬/프리팹 무수정 원칙** — 전 수정이 코드-온리, 직렬화 필드/클래스명 무변경으로
   팀원 프리팹 배선과 병합 안전성을 보존.
